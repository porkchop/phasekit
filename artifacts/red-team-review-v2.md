## Blocker resolution audit

- **F1 — `--self-check` falsifiability.** **Resolved schema-level.** The `ignore:` glob list is constrained at runtime (§8): a violation of the prefix whitelist (`docs/`, `.claude/`, `scripts/`, `templates/`, `.devcontainer/`, `capabilities/`) makes `--self-check` exit 1 and name the offending glob/path. Falsifiable test pair (positive: three named files; negative: CI-injected `docs/UNCLASSIFIED.md`) is in §8 and is acceptance criterion #2. The `--self-check` triple-state criterion (must fail on master today, pass after Slice A, fail again if `ignore: ["docs/**"]` is added) is in §8. Implementation following v2 cannot reach the gameable state without explicit assertion failure.

- **F2 — Rendered-template ownership for upgrades.** **Resolved schema-level.** New class `bootstrap-with-template-tracking` in §2; manifest schema in §3 includes `rendered_from` and `template_sha` per file (worked example shown for `.claude/CLAUDE.md`). §6 specifies advisory drift on `template_sha` change; never auto-overwritten; `--upgrade --rerender PATH` is reserved for Slice C. Falsifiable: a v2 scaffold bumping `CLAUDE.template.md` will produce a non-zero `--check --include-templates` output for projects whose manifest carries the v1 template_sha. Concern noted but not blocking: the memo doesn't include an explicit acceptance test for this in F10's 10 criteria — the schema field is locked but the advisory-on-bump behavior is only verified transitively. See "New issues" #1.

- **F3 — Collision-novel paths.** **Resolved schema-level.** New `(d) collision-novel` upgrade state in §6 with default exit code 3, `--adopt PATH` and `--rename-local PATH=NEWPATH` resolution flags in the §5 CLI table, and acceptance criterion #4 names a specific synthetic test (`docs/RUNBOOK.md` already present, scaffold v2 declares it; expect exit 3 naming both the drifted and collision-novel paths). Falsifiable and not reachable as silent overwrite if implemented as written.

- **F7 — Customization stranding via M9.4 deferral.** **Resolved schema-level.** `overlays: []` is a reserved field in the v1 manifest schema (§3, §7), present even when empty (acceptance criterion #8); `*.project.md` is reserved by being skipped in `--check`/`--upgrade` (§3); `--keep-local` is documented as a stop-gap. M9.4 can populate overlays without a `schema_version` bump. Falsifiable through criterion #8.

**No re-blockers.** All four BLOCKERs are schema-anchored, not prose-only.

## Class appendix

- **Body↔appendix consistency.** Spot-checked the files named in the body: `docs/QUALITY_GATES.md` (`scaffold` ✓), `templates/CLAUDE.template.md` (`scaffold-template` ✓), `LICENSE` (`scaffold-internal` ✓), `.claude/settings.json` (`bootstrap-with-template-tracking` — see below), `scripts/enrich-project.py` (`scaffold` with sha-disagreement warning ✓), `docs/RUNBOOK.md` (used as a hypothetical, not in appendix — fine, scaffold doesn't ship it today). No conflicts found.

- **`bootstrap-frozen` vs `bootstrap-with-template-tracking` boundary.** Clean. `bootstrap-frozen` lists only the four downstream-only docs (`SPEC.md`, `ARCHITECTURE.md`, `PHASES.md`, `PROD_REQUIREMENTS.md`); `bootstrap-with-template-tracking` lists only `.claude/settings.json`. Note that the templates that will produce these (`templates/spec.template.md`, `architecture.template.md`, `prod-requirements.template.md`) are correctly classified `scaffold-template`; the rendered downstream copies (e.g. downstream `docs/SPEC.md`) are not in the appendix because the appendix enumerates the scaffold's `git ls-files`, not downstream artifacts. This is consistent but worth a one-line note in §A.

- **`docs/SPEC.md` and friends classified `bootstrap-frozen` in the scaffold repo.** This conflation is real but not harmful for Slice A. In the scaffold repo these files are scaffold-internal (they describe the scaffold itself); downstream they are bootstrap-frozen (one-shot rendered from templates). Listing them as `bootstrap-frozen` in the scaffold appendix conflates two semantically distinct states. **Why it's fine for Slice A:** `--self-check` only needs each path to land in *exactly one* class for the scaffold-side audit; the appendix is the seed for the downstream manifest, not the scaffold's own self-classification. **Why a one-line commitment is needed:** if M9.4+ ever reads the appendix as authoritative for scaffold-side semantics, this collision will silently contaminate. Recommend §A get a one-line note: "classes describe downstream behavior; in the scaffold repo, `bootstrap-*` files are also scaffold-internal in that they're never re-installed into the scaffold itself." See Q2.

- **`.claude/settings.json` classed `bootstrap-with-template-tracking`.** **The misfit is real but recoverable.** `bootstrap-with-template-tracking` requires a `rendered_from` + `template_sha` pair. There is no `templates/settings.template.json` today; `enrich-project.py` line 230 copies `.claude/settings.json` verbatim. The class is currently *aspirational* for this file — the template doesn't exist yet. This means either: (a) Slice B must ship `templates/settings.template.json` to make the manifest schema honest, or (b) the file should be `scaffold` until then with an explicit "drift-expected" annotation, or (c) a 6th class (`scaffold-with-expected-drift`) is needed. Recommend (a) bundled with Slice B's `templates/AGENTS.template.md` work, since both are tracked the same way. Without commitment, Slice A schema lands with one entry whose `template_sha` is null — a footgun.

- **`.scaffold/manifest.json` is correctly excluded from scaffold `git ls-files` and labeled `project-committed` in the appendix.** §9 row 10 mandates commit and warns when gitignored. Internally consistent. The committed-downstream story is sound: the scaffold ships nothing under `.scaffold/`, downstream creates it on first enrichment, the engine treats absent-`.scaffold/` as a synthetic v0 reconcile case. Confirmed coherent.

## Newly surfaced questions

- **Q1 — `capabilities/project-capabilities.yaml` extension by downstream (filed as M9.6).** Deferral cost is **low**. Slice A extends this YAML with `ownership:`, `files:`, `ignore:`, `text_default:`. As long as those keys are top-level and additive, M9.6 can layer downstream-extension semantics without a schema break. Risk: if Slice A ever embeds downstream-specific data in this YAML (it shouldn't — that lives in `.scaffold/manifest.json`), the boundary blurs. Minimal commitment for v2: one sentence in §4 stating that `capabilities/project-capabilities.yaml` is scaffold-only authority and downstream extensions live in `.scaffold/manifest.json` overlays (M9.6). No new blocker.

- **Q2 — `docs/SPEC.md` etc. as `bootstrap-frozen` despite being scaffold-internal in the scaffold repo.** Deferral cost is **low** if §A gets the one-line clarification recommended above. The class semantics are downstream-facing; ambiguity only matters if a future tool treats the appendix as bidirectional. Locking in `bootstrap-frozen` doesn't foreclose later refinement — adding a `scope:` qualifier (e.g. `scope: downstream-only`) is additive. No new blocker.

- **Q3 — `examples/` as `ignore` vs `scaffold-internal`.** Deferral cost is **low**. `examples/**` is whitelisted in the `ignore:` allowed targets (§8). Switching to `scaffold-internal` later would be a cleanup, not a migration — `--self-check` would still pass. The current call (ignore) is defensible: examples are documentation-adjacent generated content that some tooling may regenerate. No commitment needed.

## New issues

- **NI-1 — F2 lacks a dedicated acceptance test.** F10's 10 criteria cover overlays (#8), migrations (#9), collision-novel (#4), atomic copy (#5), self-check controls (#1, #2), reconcile (#3), keep-local (#6), uninstall (#7), concurrency (#10) — but no criterion verifies that a `template_sha` change in the scaffold produces an advisory `--check --include-templates` signal downstream. Schema is locked, but behavior is only transitively verified. Recommend adding an 11th criterion (or amending #6): "modify `templates/CLAUDE.template.md` in scaffold; re-run `--check --include-templates` on a downstream project; advisory exit code, named path, no auto-overwrite." Non-blocking but easy to add now.

- **NI-2 — `.claude/settings.json` template absence.** Per the class-appendix discussion above. Schema commits the file to `bootstrap-with-template-tracking` semantics; Slice A will write a manifest with `template_sha: null` for this file unless a template is materialized. Recommend explicit commitment in §10 Slice B that `templates/settings.template.json` ships alongside `templates/AGENTS.template.md`. Non-blocking but a Slice B sequencing item.

- **NI-3 — `--strict` semantics on `bootstrap-*` classes is unspecified.** §3 says `sha256_strict` is "used by `--check --strict`," but `bootstrap-frozen` files are "never re-checked" (§2 table). Does `--check --strict` re-check them anyway, or remain silent? Edge case unlikely to matter in practice but the spec is ambiguous. One-line clarification would close it.

All three are memo edits, not re-plans. Slice A may begin.
